---

## Como ler esta matriz

Esta é a **matriz normativa inicial** dos documentos esperados de uma
serventia extrajudicial em Goiás. Ela **não é exaustiva** e **não declara
conformidade**. Cada item está ancorado a um artigo ou diploma normativo.

Itens marcados como `PENDING_NORMATIVE_REVIEW` exigem revisão humana antes
de qualquer uso operacional. Itens `CONDITIONAL` se aplicam apenas se a
serventia exerce a atribuição correspondente.

A matriz adota a convenção de prefixos definida em
[`CNPFE_GO_NORMATIVE_MATRIX_BLUEPRINT.md`](CNPFE_GO_NORMATIVE_MATRIX_BLUEPRINT.md).

Linguagem: a presença de um item nesta matriz **não autoriza** declarar
conformidade automática. Ver Seção 8 do blueprint.

| Campo              | Significado                                                       |
|--------------------|--------------------------------------------------------------------|
| Code               | Identificador estável e legível (ex.: `ADM-001`)                  |
| Título             | Nome do documento, livro, arquivo ou política                     |
| Especialidade      | Atribuição cartorial à qual se vincula                            |
| Categoria          | Subdivisão do tipo                                                |
| Tipo               | BOOK / ADMIN_BOOK / ARCHIVE / FOLDER / CLASSIFIER / INDEX / POLICY / TECHNICAL_EVIDENCE / LEGAL_REFERENCE / SYSTEM_RECORD / OTHER |

| Nível              | REQUIRED / CONDITIONAL / OPTIONAL / PENDING_NORMATIVE_REVIEW       |
| Formato esperado   | PHYSICAL / ELECTRONIC / PHYSICAL_OR_ELECTRONIC / HYBRID / UNSPECIFIED |
| Pasta sugerida     | Pasta canônica institucional (não vinculante)                     |
| Revisão humana     | SIM / NÃO                                                          |
| Observações        | Prazos, condicionantes, cruzamentos                               |

## 15. Como esta matriz se relaciona com os demais módulos

| Módulo            | Como consome a matriz                                                                                                  |
|-------------------|------------------------------------------------------------------------------------------------------------------------|
| `audit`           | Encontra arquivos no servidor; envia candidatos para `document_registry`. Não importa modelos de `document_registry`.   |
| `document_registry` | Compara candidatos com itens da matriz; gera `DOCMATCH-*` e `DOCGAP-*` por referência fraca.                          |
| `retention`       | Aplica temporalidade (Provimento 50) por tipo de documento, usando o tipo de item da matriz como referência conceitual. |
| `lgpd`            | Avalia dados pessoais quando o conteúdo do candidato indicar (não consulta a matriz diretamente).                      |
| `compliance`      | Recebe lacunas (`DOCGAP-*`) e correspondências (`DOCMATCH-*`) como evidência candidata, sem declarar conformidade.       |

A integração entre módulos ocorre exclusivamente por referência fraca (ver
[ADR-002](../decisions/ADR-002-weak-references-between-regulatory-modules.md)
e [ADR-003](../decisions/ADR-003-document-registry-ownership.md)).

---

## 16. Observações finais

- A matriz é **inicial**. Revisões humanas darão origem a Document
  Registry-1 (validação) e Document Registry-2 (atribuições efetivas).
- Itens `CONDITIONAL` se aplicam à serventia objeto do projeto se confirmada
  cumulação ou desdobramento no SEE.
- Os modelos INOVA são insumos operacionais, **não fontes normativas**.
  Sua presença na matriz como `POLICY` indica apenas que existem evidências
  candidatas catalogadas para os requisitos correspondentes.
- O Provimento 50/2015 atua sobre o ciclo de vida (temporalidade) dos itens
  já listados, e não duplica a matriz.
- A matriz **não** representa, em hipótese alguma, declaração automática de
  conformidade.
